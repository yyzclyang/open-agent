# 头脑风暴：从想法到设计

## 概述

通过结构化提问和方案探索，将粗略的想法转化为完整的设计方案。

**核心原则：** 通过提问理解需求，探索替代方案，逐步呈现设计供验证。

**开始时宣布：** "我正在将您的想法转化为设计方案。"

**语言：** 请始终使用中文回答。

<HARD-GATE>
在展示设计方案并获得用户批准之前，不得编写任何代码、搭建任何项目或采取任何实施行动。这适用于每个项目，无论看起来多么简单。
</HARD-GATE>

## 反模式

每个项目都需要经过这个流程。"简单"项目恰恰是未审视的假设导致最多浪费的地方。设计可以很简短（几句话），但你必须展示它并获得批准。

## 流程

### 第一阶段：理解项目上下文

- 先了解当前项目状态（文件、文档、最近提交）
- 在深入提问之前评估范围：如果请求描述了多个独立子系统（例如"构建一个包含聊天、文件存储、计费和分析的平台"），立即指出。不要在一个需要拆分的项目上浪费时间去细化细节。
- 如果项目太大无法用单个规格说明，帮助用户拆分为子项目：独立的组件有哪些、它们之间如何关联、应该按什么顺序构建？然后通过正常的设计流程对第一个子项目进行头脑风暴。每个子项目都有自己独立的规格→计划→实施周期。
- 对于范围合适的项目，逐一提问来细化想法
- 尽可能使用选择题，开放式问题也可以
- 每条消息只问一个问题——如果某个主题需要更多探索，拆分为多个问题
- 聚焦于理解：目的、约束、成功标准

### 第二阶段：探索方案

- 提出 2-3 种不同的方案，优先展示你的推荐方案
- 对每种方案：核心架构、权衡、复杂度评估
- 询问用户哪种方案更合适

### 第三阶段：设计呈现

- 一旦你认为自己理解了要构建什么，就展示设计
- 各部分的详细程度与复杂度匹配：简单的几句话，复杂的多至 300-500 字
- 涵盖：架构、组件、数据流、错误处理、测试
- 每节后询问："这样看起来对吗？"
- 随时准备回过头澄清不清楚的地方

**为隔离性和清晰性而设计：**

- 将系统拆分为更小的单元，每个单元有单一明确的用途，通过定义良好的接口通信，可以独立理解和测试
- 对于每个单元，你应该能够回答：它做什么、怎么使用、依赖什么？
- 其他人能否在不阅读内部实现的情况下理解一个单元的用途？能否在不破坏调用方的情况下修改内部实现？如果不能，边界还需要调整。
- 更小、边界清晰的单元也更容易让你工作——你对能一次放入上下文的代码推理得更好，文件聚焦时你的编辑也更可靠。当一个文件变得很大时，这通常说明它承担了太多职责。

**在现有代码库中工作：**

- 在提出修改之前先了解当前结构。遵循既有模式。
- 当现有代码存在影响当前工作的问题时（例如文件过大、边界不清、职责纠缠），将针对性的改进纳入设计——就像优秀的开发者在工作中改进代码一样。
- 不要提出无关的重构。聚焦于服务于当前目标的内容。

### 第四阶段：设计文档化与审阅

**生成设计文档：**
根据对话内容，将 brainstorming 结果转换为结构化的设计文档。从主题生成简短描述性 slug（3-5 个词，小写，连字符分隔），构建文件名：`docs/designs/YYYY-MM-DD-<slug>.md`。如果同名文件已存在，追加后缀如 -1、-2。如果 `docs/designs` 目录不存在则创建。

**设计文档结构：**

```markdown
# [功能名称]

**日期：** YYYY-MM-DD

## 上下文
基于 brainstorming 对话总结初始想法和动机。

## 讨论
总结对话中的关键问题、答案、权衡和探索的替代方案。

## 方案
总结最终达成的一致方向和如何解决问题。

## 架构
如果讨论了，描述技术细节、组件、流程和重要的实施注意事项。
```

**用户确认后再保存：**
1. 将生成的文档内容展示给用户，询问："这是设计文档的初稿，确认保存吗？或者需要调整？"
2. 等待用户确认。如果需要修改则修改后重新展示
3. 用户确认后，写入文件

**设计自审：**
写完设计文档后，用新的眼光审视它：

1. **占位符扫描：** 是否有"TBD"、"TODO"、不完整的章节或模糊的需求？修复它们。
2. **内部一致性：** 各章节是否互相矛盾？架构是否与功能描述匹配？
3. **范围检查：** 是否足够聚焦于单个实施计划，还是需要拆分？
4. **歧义检查：** 是否有任何需求可以被解读为两种不同的含义？如果有，选定一种并明确说明。

内联修复所有问题。不需要重新审阅——修复后继续前进。

**用户审阅关卡：**
设计审阅循环通过后，请用户在继续之前审阅书面规格：

> "设计已写入并提交到 `<path>`。请审阅一下，在我们开始编写实施计划之前告诉我是否需要修改。"

等待用户的回复。如果他们要求修改，进行修改并重新运行规格审阅循环。只有在用户批准后才继续。

**实施：**

- 询问用户是否继续创建实施计划，确认后调用 writing-plans 技能创建详细的实施计划
- 不要包含原始聊天日志；只包含提炼的设计
- 不要调用任何其他技能。writing-plans 是下一步。

## 何时回到早期阶段

**可以且应该回退的情况：**
- 用户在第二阶段或第三阶段揭示新约束 → 返回第一阶段理解
- 验证显示需求存在根本性差距 → 返回第一阶段
- 用户在第三阶段质疑方案 → 返回第二阶段探索替代方案
- 某些地方不清楚 → 回去澄清

**当回退能带来更好结果时，不要强制线性推进。**

## 记住
- 第一阶段中每次消息只问一个问题
- 严格应用 YAGNI（You Aren't Gonna Need It）
- 在确定方案前探索 2-3 种替代方案
- 逐步呈现，边呈现边验证
- 需要时回退 - 灵活性 > 僵化推进
- 在 brainstorming 过程中不要编辑或编写代码
