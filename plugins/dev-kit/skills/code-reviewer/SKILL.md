# 代码审阅

当实现了一个完整功能块、计划中的某个步骤或一组相关改动后，使用此技能进行结构化审阅。

## 核心目标

- 对照原始计划、设计或需求核对实现是否偏离
- 评估代码质量、错误处理、可维护性与潜在风险
- 输出清晰、可执行的修复建议

## 审阅流程

1. 收集审阅上下文：原始计划、需求说明、相关改动和受影响文件。
2. 将当前实现与原始计划逐项对照，识别缺漏、偏差和额外实现。
3. 检查代码组织、命名、边界、错误处理、类型安全和防御性编程。
4. 评估测试覆盖、架构一致性、性能风险和安全问题。
5. 输出结构化结论，按严重性分类问题并给出明确建议。

## 审阅维度

### 计划对齐

- 当前实现是否覆盖原始计划中的功能点
- 是否存在未说明的设计偏移
- 偏移是合理改进还是行为回归

### 代码质量

- 是否遵循既有模式和约定
- 是否存在脆弱逻辑、重复代码或难维护结构
- 命名、抽象和边界是否清晰

### 架构与集成

- 是否符合现有架构和模块职责
- 与上下游模块的接口是否清楚且稳定
- 是否引入不必要耦合

### 验证与风险

- 测试是否覆盖关键路径
- 是否存在遗漏的异常场景
- 是否存在性能、安全或兼容性风险

## 输出要求

- 先列问题，再给简短结论
- 问题按严重性分类：关键、重要、建议
- 每个问题都应包含具体位置、原因和建议
- 如果没有发现问题，明确说明未发现显著问题，并指出剩余风险或验证缺口
