# 编写计划

## 概述

编写详尽的实施计划，假设执行工程师对代码库毫无了解，品味也存疑。记录他们需要知道的一切：每个任务要修改哪些文件、代码、测试、可能需要查阅的文档、如何测试。将整个计划拆分为可小口吞咽的任务。DRY。YAGNI。TDD。频繁提交。

假设他们是有能力的开发者，但几乎不了解我们的工具集或问题领域。假设他们不太擅长测试设计。

**开始时宣布：** "我正在使用 writing-plans 技能创建实施计划。"

**上下文：** 应在专用 worktree 中运行（由 brainstorming 技能创建）。

**计划保存到：** `docs/plans/YYYY-MM-DD-<feature-name>.md`
- （用户对计划位置的偏好覆盖此默认值）

## 范围检查

如果规格涵盖多个独立子系统，应该在头脑风暴阶段就已拆分为子项目规格。如果没有，建议拆分为独立的计划——每个子系统一个。每个计划应能独立产出可工作、可测试的软件。

## 文件结构

在定义任务之前，先梳理需要创建或修改哪些文件，以及每个文件的职责。分解决策在此锁定。

- 设计边界清晰、接口定义良好的单元。每个文件应有单一明确的职责。
- 你对能一次放入上下文的代码推理得更好，文件聚焦时编辑也更可靠。优先选择小而聚焦的文件，而非大而臃肿的文件。
- 一起变更的文件应该放在一起。按职责拆分，而非按技术层拆分。
- 在现有代码库中，遵循既有模式。如果代码库使用大文件，不要单方面重构——但如果你要修改的文件已经过于臃肿，在计划中加入拆分是合理的。

此结构指导任务分解。每个任务应产出独立可理解的变更。

## 小口吞咽的任务粒度

**每个步骤是一个动作（2-5 分钟）：**
- "编写失败的测试" — 一个步骤
- "运行测试确认它失败" — 一个步骤
- "编写最小代码使测试通过" — 一个步骤
- "运行测试确认通过" — 一个步骤
- "提交" — 一个步骤

## 计划文档头部

**每个计划必须以此头部开始：**

```markdown
# [功能名称] 实施计划

> **给代理工作者：** 必备子技能：使用 dev-kit:subagent-driven-development（推荐）或 dev-kit:executing-plans 逐任务实施此计划。步骤使用复选框（`- [ ]`）语法进行追踪。

**目标：** [一句话描述要构建什么]

**架构：** [2-3 句话描述方案]

**技术栈：** [关键技术/库]

---
```

## 任务结构

````markdown
### 任务 N：[组件名称]

**文件：**
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：编写失败的测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试确认失败**

运行：`pytest tests/path/test.py::test_name -v`
预期：失败，提示 "function not defined"

- [ ] **步骤 3：编写最小实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试确认通过**

运行：`pytest tests/path/test.py::test_name -v`
预期：通过

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: 添加特定功能"
```
````

## 禁止占位符

每个步骤必须包含工程师所需的实际内容。以下是**计划缺陷**——永远不要写这些：
- "TBD"、"TODO"、"以后实现"、"补充细节"
- "添加适当的错误处理" / "添加验证" / "处理边界情况"
- "为上面的代码编写测试"（没有实际测试代码）
- "与任务 N 类似"（重复代码——工程师可能不按顺序阅读任务）
- 只描述做什么而不展示怎么做的步骤（代码步骤必须有代码块）
- 引用未在任何任务中定义的类型、函数或方法

## 记住
- 始终提供精确的文件路径
- 每个步骤包含完整代码——如果步骤修改了代码，展示代码
- 精确的命令和预期输出
- DRY、YAGNI、TDD、频繁提交

## 自审

写完完整计划后，用新的眼光审视规格，将计划与规格对照检查。这是你自己运行的检查清单——不是派发子代理。

**1. 规格覆盖：** 快速浏览规格中的每个章节/需求。你能指出哪个任务实现了它吗？列出任何遗漏。

**2. 占位符扫描：** 在计划中搜索危险信号——上述"禁止占位符"部分中的任何模式。修复它们。

**3. 类型一致性：** 后续任务中使用的类型、方法签名和属性名是否与早期任务中定义的匹配？任务 3 中叫 `clearLayers()` 但任务 7 中叫 `clearFullLayers()` 就是 bug。

如果发现问题，内联修复。不需要重新审阅——修复后继续。如果发现规格需求没有对应任务，添加任务。

## 执行交接

保存计划后，提供执行选项：

**"计划已完成并保存到 `docs/plans/<filename>.md`。两种执行方式：**

**1. 子代理（推荐）** - 我为每个任务派发一个新的子代理，任务间审阅，快速迭代

**2. 内联执行** - 在当前会话中使用 executing-plans 执行任务，带检查点的批量执行

**选哪种方式？"**

**如果选择子代理：**
- **必备子技能：** 使用 dev-kit:subagent-driven-development
- 每个任务一个新子代理 + 两阶段审阅

**如果选择内联执行：**
- **必备子技能：** 使用 dev-kit:executing-plans
- 带检查点的批量执行
