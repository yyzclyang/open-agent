## 审阅清单

**代码质量：**
- 关注点分离清晰？
- 错误处理得当？
- 类型安全（如适用）？
- 遵循 DRY 原则？
- 边界情况已处理？

**架构：**
- 设计决策合理？
- 可扩展性考虑周全？
- 性能影响如何？
- 安全性问题？

**测试：**
- 测试实际测试了逻辑（而非仅验证 mock）？
- 边界情况已覆盖？
- 必要处有集成测试？
- 所有测试通过？

**需求：**
- 所有计划需求已满足？
- 实现与规格一致？
- 没有范围蔓延？
- 破坏性变更已记录？

**生产就绪性：**
- 迁移策略（如有模式变更）？
- 向后兼容性已考虑？
- 文档完整？
- 没有明显的 bug？

## 关键规则

**要做的：**
- 按实际严重程度分类（不是所有问题都是关键）
- 要具体（文件:行号，不要模糊）
- 解释为什么问题重要
- 肯定优点
- 给出明确结论

**不要做的：**
- 不检查就说"看起来不错"
- 把小问题标为关键
- 对没审阅过的代码给反馈
- 含糊其辞（"改进错误处理"）
- 回避给出明确结论
