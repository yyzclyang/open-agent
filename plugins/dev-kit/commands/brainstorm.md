# 将想法转化为设计方案

## 概述

通过结构化提问和方案探索，将粗略的想法转化为完整的设计方案。

**核心原则：** 通过提问理解需求，探索替代方案，逐步呈现设计供验证。

**开始时宣布：** "我正在将您的想法转化为设计方案。"

**语言：** 请始终使用中文回答。

## 流程

### 第一阶段：理解

**首先：** 根据用户提供的关键字/需求，使用代码搜索能力查找相关代码、术语定义和业务逻辑，建立上下文理解。

- 一次只问一个问题来细化想法
- 尽可能使用选择题
- 收集：目的、约束条件、成功标准

### 第二阶段：探索
- 提出 2-3 种不同的方案
- 对每种方案：核心架构、权衡、复杂度评估
- 询问您的合作伙伴哪种方案更合适

### 第三阶段：设计呈现
- 以 200-300 字的小节呈现
- 涵盖：架构、组件、数据流、错误处理、测试
- 每节后询问："这样看起来对吗？"

## 何时回到早期阶段

**可以且应该回退的情况：**
- 合作伙伴在第二阶段或第三阶段揭示新约束 → 返回第一阶段理解
- 验证显示需求存在根本性差距 → 返回第一阶段
- 合作伙伴在第三阶段质疑方案 → 返回第二阶段探索替代方案
- 某些地方不清楚 → 回去澄清

**当回退能带来更好结果时，不要强制线性推进。**

## 记住
- 第一阶段中每次消息只问一个问题
- 严格应用 YAGNI（You Aren't Gonna Need It）
- 在确定方案前探索 2-3 种替代方案
- 逐步呈现，边呈现边验证
- 需要时回退 - 灵活性 > 僵化推进
- 在 brainstorming 过程中不要编辑或编写代码
